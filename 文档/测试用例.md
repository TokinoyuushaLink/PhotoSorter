## 二、逻辑链路覆盖

### 2.1 归类路径分支

| 场景 | 路径 | 验证要点 |
|------|------|---------|
| 普通归类（未归类视图，单张，键盘） | `assignToAlbum` → `photosStore.addIDs` → `sortHistory.record(.classify)` | undoStack 增加 1 条，网格消失，已分类相册计数 +1 |
| 普通归类（多选） | 同上，ids 为多个 | undoStack 增加 1 条（批量打包），批量消失 |
| 单图模式归类 | 路由到 `singleModeAssets[currentIndex].id` | 归类当前图，队列自动跳下一张，singleModeCurrentIndex 正确 |
| 归类已在其他相册的照片 | 先从旧相册移除 → 再 classify 到新相册 | 旧相册分组消失或减少，新相册增加；undoStack 只记录新 classify |
| 归类在待删除中的照片 | 先 restoreFromPendingDelete → 再 classify | 待删除分组减少，新相册增加；两步均可撤销 |

### 2.2 撤销链路完整性

| 场景 | undoStack 操作 | 预期状态 |
|------|--------------|---------|
| classify → undo | popUndo(.classify) → restoreAssets → pushRedo | 照片回未归类，redo 可重做 |
| classify → undo → redo | popRedo → re-classify | 照片再次归类，undoStack 恢复 |
| classify → undo → 新 classify | redoStack 清空 | 旧 redo 丢失，只能 undo 新操作 |
| moveToPendingDelete(from album) → undo | popUndo(.moveToPendingDelete) → removeFromPending → re-classify 到原相册 | 照片从待删除回到原相册分组 |
| restoreFromPendingDelete → undo | popUndo(.restoreFromPendingDelete) → addPendingDelete | 照片重回待删除 |
| 连续 10 次归类 → 连续 10 次 undo | 每次 undo 弹出对应 action | 顺序严格 LIFO，最终 undoStack 为空 |

### 2.3 已分类视图数据一致性

| 场景 | 验证要点 |
|------|---------|
| 归类 N 张到相册 A，M 张到相册 B | 已分类视图显示 2 个分组，A 有 N 张、B 有 M 张 |
| 撤销其中 1 张 A 的归类 | A 分组减少 1 张，若 A 变 0 则分组消失 |
| 同一张照片重复归类到不同相册 | 只出现在最新的相册中，旧相册不重复显示 |
| 有待删除 + 有归类 | 已分类视图末尾追加"待删除"分组，其余相册不受影响 |
| commitAll 后 | undoStack 清空，已分类视图变空，showClassifiedView 自动切回 false |

### 2.4 单图队列完整性

| 场景 | 验证要点 |
|------|---------|
| 未归类视图无选中时按 Space | singleModeAssets = photosStore.assets 全量 |
| 未归类视图有多选时按 Space | singleModeAssets = 选中子集，按 assets 顺序 |
| 已分类视图某分组中按 Space | singleModeAssets = 该分组照片 |
| 单图下归类当前照片 | 当前图从队列移除，singleModeCurrentIndex 指向下一张；若为最后一张则退出单图 |
| 单图队列只剩 1 张时归类 | 单图模式退出，回到网格 |
| 单图下左右翻页 | currentIndex 边界不越界（首张左划无效，末张右划无效）|

## 三、性能与功耗

### 3.1 缩略图加载（网格）

| 场景 | 观察项 | 可接受基准 |
|------|--------|----------|
| 启动后加载 500 张照片 | 网格首屏渲染时间，CPU 占用曲线 | 首屏格子 < 1s 出现，CPU 峰值后回落 |
| 快速向下滚动（inertia scroll）| Cell 缓存命中率，白格/占位格比例 | 大部分格子有缩略图，偶尔短暂占位可接受 |
| 滚动停止后 | CPU 是否回落到接近 0% | 静止状态下 CPU < 5% |
| ThumbnailCache 400 张上限触发 LRU 淘汰 | 淘汰后重新滚回该区域，缩略图重新加载 | 无崩溃，加载略延迟但正确显示 |
| 库中有 2000+ 张照片 | 内存占用峰值 | 缩略图缓存不超过约 200MB（400 张 × ~500KB 估算）|

### 3.2 预览图加载（单图）

| 场景 | 观察项 | 可接受基准 |
|------|--------|----------|
| 进入单图，当前图无缓存 | 全尺寸加载耗时，是否有白屏/低质量过渡 | < 500ms 显示低质量图，全尺寸在 2s 内替换 |
| 连续快速翻页 10 张 | 内存增长趋势，是否有泄漏 | 内存不持续增长，翻页后旧图内存可回收 |
| 视频进入单图 | AVPlayer 初始化时间，帧率 | < 1s 开始播放，帧率稳定 |
| GIF 进入单图 | CPU 占用（帧解码） | GIF 播放 CPU < 20%，不影响主线程 |
| 相邻帧预加载 | 翻到下一张时是否已有 preview | 大多数情况下翻页即时显示，无白屏 |

### 3.3 归类操作响应

| 场景 | 观察项 | 可接受基准 |
|------|--------|----------|
| 按数字键归类单张 | 从按键到网格格子消失的延迟 | < 50ms，感觉即时 |
| 多选 50 张后一次性归类 | 批量移除动画，CPU 峰值 | < 200ms 完成差分更新，无卡顿 |
| commitAll 100 张到 3 个相册 | PHPhotoLibrary 写入耗时 | 进度提示友好，完成后 < 5s |

### 3.4 各视图刷新功耗

| 视图/交互 | 观察项 | 期望行为 |
|----------|--------|---------|
| 网格静止状态 | Energy Impact（Activity Monitor）| 极低，无定时刷新 |
| 单图模式静止（非视频/GIF）| CPU / GPU 占用 | 接近零，无不必要重绘 |
| 视频单图播放中 | CPU + GPU 占用 | GPU 解码，CPU 保持低位 |
| GIF 单图播放中 | CPU 占用 | 帧解码占用可接受，不应超过单核 30% |
| 右侧面板展开/收起动画 | 动画帧率 | 60fps 流畅，无掉帧 |
| 收藏条悬停自动显隐动画 | 重绘范围 | 只刷新收藏条区域，网格不重绘 |
| 拖拽调整面板宽度 | 实时响应，CPU 占用 | 跟手流畅，释放后占用回落 |
